# Spec Delta: create-new-repo

## MODIFIED Requirements

### Requirement: Deterministic repo-creation engine

The system SHALL provide a deterministic script `scripts/create_beelink_repo.sh`
that provisions a brand-new beelink repo with NO LLM in the critical path. Given a
bare `--name`, it SHALL validate the name against
`^[a-z0-9]([a-z0-9.-]*[a-z0-9])?$` — lower-case alphanumerics with internal dots
and hyphens, never a leading or trailing separator, so domain-style names such as
`foo.ru` and `example.ru` are accepted while `Foo_Bar`, `-leading`,
`UPPER`, `a/b` and `has space` are rejected. It SHALL refuse to clobber an
existing dir or GitHub repo, scaffold the repo to beelink standards, create a
private GitHub repo via API+PAT (no `gh`), push over SSH, and print `{url, path}`
on success. Every external binary and path SHALL be an injectable env seam so all
I/O can be mocked offline. Any step failing SHALL exit non-zero.

The generated `docker-compose.yml` SHALL be runnable as-is on the beelink server:
it SHALL reference the repo-root env file via `env_file: - ../.env` (the compose
lives under `deploy/`) and SHALL carry the egress-proxy `environment:` block so a
new service can reach the blocked APIs (Claude / GitHub / OpenAI / Telegram) that
the host only routes through `xray-client`.

A step failing AFTER the target directory is created SHALL NOT leave that directory
behind: the script SHALL remove the freshly-created `$DEST` on failure so a
transient error (deployer down, push rejected) does not permanently block a retry.

#### Scenario: Invalid name writes nothing

- **GIVEN** the script is invoked with an invalid bare name such as `Foo_Bar`
- **WHEN** it runs
- **THEN** it exits non-zero
- **AND** no new directory is created under the projects root

#### Scenario: A domain-style dotted name is accepted

- **GIVEN** the script is invoked with a bare name containing internal dots such
  as `foo.ru`
- **WHEN** it runs (curl/git/deployer seams stubbed offline)
- **THEN** it exits zero
- **AND** it provisions `beelink-foo.ru` and prints the `{url, path}` JSON

#### Scenario: Refuses to clobber an existing dir or repo

- **GIVEN** the target dir already exists OR a `GET /repos/Garbuzenko/beelink-<name>`
  (via the injected curl) returns a code other than 404
- **WHEN** the script runs
- **THEN** it exits non-zero
- **AND** it performs no deployer registration, no GitHub creation and no push

#### Scenario: Template copy excludes .env.example and writes a real .env

- **GIVEN** a template directory and a mocked deployer response
- **WHEN** the script copies the template and writes the env file
- **THEN** the copied tree contains no `.env.example`
- **AND** a `.env` exists containing `PROJECT_NAME=beelink-<name>`
- **AND** `.env` is matched by the repo `.gitignore` (not staged in the initial commit)

#### Scenario: Generated docker-compose is standards-compliant and reachable

- **WHEN** the script writes `deploy/docker-compose.yml`
- **THEN** it sets `container_name: beelink-<name>`
- **AND** it exposes no host ports
- **AND** it includes a resource limit (under `deploy.resources.limits`)
- **AND** it includes a `healthcheck`
- **AND** it references `env_file: - ../.env`
- **AND** it sets the egress proxy via an `environment:` block pointing at
  `http://xray-client:8080`

#### Scenario: A failed run leaves no orphaned target directory

- **GIVEN** the target dir and GitHub repo are free, but a later step fails (e.g.
  the deployer registration or the push)
- **WHEN** the script exits non-zero
- **THEN** the freshly-created target directory has been removed
- **AND** a subsequent re-run is not blocked by a leftover directory

#### Scenario: Vision file and OpenSpec scaffolding

- **GIVEN** a `--vision` argument
- **WHEN** the script runs
- **THEN** `docs/strategy/vision.md` is written whose first line is the AUTOGENERATED
  header `<!-- AUTOGENERATED FROM GOOGLE SHEET — edit in the sheet, not here -->`
- **AND** an `openspec/` directory exists in the new repo so the OpenSpec gate passes

#### Scenario: GitHub creation and push are deterministic

- **WHEN** the script creates and pushes the repo
- **THEN** it reads the PAT from `$HOME/.config/gh-token`
- **AND** it POSTs to `https://api.github.com/user/repos` with a Bearer token
- **AND** it adds the SSH remote `git@github.com:Garbuzenko/beelink-<name>.git`
- **AND** it pushes via SSH (never an HTTPS/live call in tests)
